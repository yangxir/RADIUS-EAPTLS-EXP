    üì° RADIUS Proxy Server Authentication Guide¶

    üìå 1. Role of the Proxy Server¶

    üîπ Why is a Proxy Server Needed?¶

    In the RADIUS authentication architecture, Proxy Server is mainly used to forward authentication requests rather than processing authentication directly. Its core role is to ensure that client authentication requests can reach the final RADIUS server in a distributed environment.

    Proxy Server is essential in the following scenarios: - Enterprise WiFi authentication (e.g., Eduroam multi-site roaming) - Multi-level RADIUS authentication (e.g., ISP authentication systems) - Cross-national/cross-regional WiFi authentication (authentication forwarding between RADIUS servers in different countries)

    üì° 2. Proxy Server Workflow¶

    1Ô∏è‚É£ Client sends an EAP-TLS authentication request
    2Ô∏è‚É£ Proxy Server receives the request and forwards it to the final RADIUS Server
    3Ô∏è‚É£ RADIUS Server processes the authentication request and returns the result
    4Ô∏è‚É£ Proxy Server sends the result back to the client
    5Ô∏è‚É£ Client successfully connects to WiFi üéâ

    üìç 3.3 Configure Proxy Server¶

    üîπ 1. Modify proxy.conf¶

    type = radius
    authhost = 192.168.8.195:1812
    accthost = 192.168.8.195:1813
    secret = shared_secret_key
}
    ‚úÖ Ensure Proxy Server acts only as a "relay" and not a "local authentication" server

    üîπ 2. Configure RADIUS Server to Allow Proxy¶

    Add Proxy Server to clients.conf:

    secret = shared_secret_key
    shortname = proxy
}
    ‚úÖ Ensure the secret is consistent between Proxy Server and RADIUS Server to avoid errors

    üîπ 3. Start RADIUS Server and Proxy Server¶

    View logs: 
    journalctl -u freeradius -f
    ‚úÖ Ensure services are running normally without errors

    üî¥ 1. Proxy Server Did Not Forward Requests Correctly¶

    ‚ùå Issue¶

    • The Proxy Server did not forward the EAP-TLS authentication request but handled authentication locally instead.
    • No authentication request forwarding was found in the radiusd -X logs.

    ‚úÖ Solution¶

    1. Enable proxy forwarding 
      proxy_requests = yes
    2. Ensure realm configuration in proxy.conf is correct 
      realm DEFAULT {
    type = radius
    authhost = 192.168.8.195:1812
    accthost = 192.168.8.195:1813
    secret = shared_secret_key
    auth_pool = my_auth_failover  # Replace with your own configuration
}
    3. Restart FreeRADIUS 
      systemctl restart freeradius
      ‚úÖ Successfully forwarded EAP-TLS authentication requests to the RADIUS Server! üéâ

    üî¥ 2. Certificate Chain Error¶

    ‚ùå Issue¶

    • eapol_test authentication failed, with logs showing: 
      Certificate chain - 1 cert(s) untrusted
    • RADIUS Server logs showed: 
      TLS Alert read: warning: unknown CA

    ‚úÖ Solution¶

    1. Ensure eap.conf on the RADIUS Server is configured correctly 
      tls-config tls-common {
    private_key_file = /etc/freeradius/3.0/certs/server.key
    certificate_file = /etc/freeradius/3.0/certs/server.pem
    ca_file = /etc/freeradius/3.0/certs/ca.pem
}
    2. Import the CA certificate on the client 
      scp root@192.168.8.195:/etc/freeradius/3.0/certs/ca.crt ~/Desktop/
      ‚úÖ Certificates successfully matched, authentication passed! üéâ
